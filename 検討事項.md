## 1. パスの扱い

- 合法手がない場合のパスは公式リバーシルールで強制。`PA\r\n` を CPU から送る
- **未確定**: 合法手があるのに CPU が `PA\r\n` を返してきた場合、PC はどう扱うか
  - 案 A: そのままパス扱い（CPU の自由意志を尊重、ただし反則負け扱いしない）
  - 案 B: 不正応答として扱い、切断または `ER` を返す
  - 案 C: 合法手チェックは PC 側で行い、不正なら黙って無視して再度手番通知
- **未確定**: PC → CPU の `PA\r\n`（相手パス通知）は、相手に合法手がない時のみ
  送られるのか、相手の自主パス（実装によってはありうる）も含むのか

## 2. 無効な着手 / 不正応答への対応

**決定: 案 B を採択** — 不正時は `ER\r\n` を返し、送信側は再試行可
（[README.md §4.1](README.md#41-仕様違反受信時の扱い) 参照）。

- 対象: `MOz9` のような範囲外座標、合法でない座標 (`MOa1` が非合法のとき等)、
  小文字コマンド、CR 混入、未知コマンド全般
- 受信側 (CPU / PC 両方) は `ER\r\n` を返す。送信側は [README.md §6.2 #7](README.md#62-cpu--pccpu-からの応答)
  に従いログ記録のみで自動再送はしない (無限ループ防止)。CPU は再度 `MO?` や `PA`
  を送り直してよい

参考: 当初は以下を検討していた。

- 案 A: PC が合法性を判定し、不正なら黙って無視して再度 `MO?` を待つ
- 案 B: 不正時は `ER\r\n` を返し、CPU 側は再試行 ← **採択**
- 案 C: 不正時は対局を中断し、反則負けとして PC 側で `EW`/`EB` を送る

## 3. 状態遷移表の未確定セル

[README.md §11. 状態遷移表](README.md#11-状態遷移表) で `※` で示したセルの扱い。

- **C2 (MY_TURN) / C3 (WAIT_OPP) 中に `SB` / `SW` を受信**: 対局中に新規対局通知が
  来た場合の挙動。PC 側の誤動作か、リセット復帰の意図か判別できない
  - 案 A: 現対局を破棄して新規対局として処理（`SB`→C2 / `SW`→C3）
  - 案 B: 無視（対局中に新規対局は論理的にありえない）
- **C2 (MY_TURN) 中に `PA` を受信**: 自分の手番中に相手パス通知はありえないが、
  防御的に無視するか `ER\r\n` を返すか

## 4. FPGA 実装向け改善候補 (v0.2 以降検討)

本書 v0.1 は FPGA/MCU 実装をおおむね想定しているが、以下の項目は大会運営開始前に
固めておきたい。優先度は 🔴 高 / 🟡 中 / 🟢 低。

### 🔴 高優先 — **v0.1 で実装済**

- ✅ **A. `VE` 応答の名前省略許可** — `VE01\r\n` を合法化（README.md §6.2 #4）
- ✅ **B. `VE` の `<NN>` フォーマット厳格化** — 2 桁 ASCII 数字固定、先頭ゼロ必須（README.md §6.2 #4）
- ✅ **C. `ER` のコード化 + 任意理由** — `ER<NN>[ <reason>]\r\n` 形式、
  [README.md §6.3](README.md#63-er-エラーコード表) にエラーコード表を新設
  (00 Generic / 01 Unknown / 02 Lowercase / 03 CR / 04 Invalid coord /
  05 Illegal move / 06 Invalid state / 30+ ユーザ定義)

### 🟡 中優先 (v0.2 で検討)

- **D. `NC<nodes>,<ms>\r\n` は実装不要と強調**
  10 進数 2 個カンマ区切りは FPGA では `printf("%d")` 相当が必要で重い。
  README.md §6.2 #9 に「FPGA 実装では省略推奨、NBoard 風 PC 実装向けの拡張」と注記

- **E. `ST<text>\r\n` の最大長と文字種を制限**
  現仕様: 長さ無制限、ASCII
  提案: 最大 64 文字、ASCII printable (0x20-0x7E) のみ。RX バッファ溢れ防止

- **F. 送受信ペイロードの許容文字種明記（README.md §4）**
  現仕様: 「ASCII」とだけ
  提案: **ASCII printable (0x20-0x7E) + CR+LF (0x0D 0x0A)** のみ許容。タブ、DEL、制御文字、
  非 ASCII バイトは §4.1 違反として ER。

### 🟢 低優先 (現状で問題なし、明記すれば親切)

- **G. RX バッファ推奨サイズ**
  現 80 バイト推奨は余裕込み、64 バイトでも実害なし
- **H. UART ビットオーダー**
  標準準拠で LSB first だが README.md §3 に明記すると親切
- **I. 半二重 / 全二重の明記**
  送受信は独立 (全二重) と README.md §3 に明記推奨
- **J. 同時アウトスタンディング応答数**
  「受信順に 1 対 1 で応答する」と明記すると PC 実装が楽

### 却下 (変更しない)

- **BO の短縮形 / デリミタ追加**: 64 バイト BRAM 1 個で足りるので不要
- **座標のバイナリ化**: ASCII 可読性を損なう、debug 不能になる
- **ボーレート固定**: FPGA 側の都合で 9600/38400/115200 選べるメリット大
